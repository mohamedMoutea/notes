Testing Checklist (Takeaways)
When hunting for OAuth bugs, test the redirect_uri parameter for these common misconfigurations:

[ ] Suffix Manipulation: Can you append a new TLD? (example.com.evil.com)

[ ] Subdomain Injection: If *.example.com is allowed, can you use attacker.example.com?

[ ] Path Traversal: Does example.com/callback/../logout work?

[ ] Weak Regex: Does it allow any string that simply contains the brand name?

[ ] Protocol Switching: Can you switch https to http or javascript:?


When a user signed up via Yahoo OAuth, Flurry needed to create a corresponding account in their own database. Instead of creating a "password-less" account, they reused their existing registration API which required a password field.

To bypass this requirement, the developers hardcoded a placeholder:

The Request: POST /auth/v1/account

The Payload: {"attributes": {"email": "user@yahoo.com", "password": "not-provided"}}

Testing Checklist (Takeaways)
When testing OAuth or Third-Party integrations, look for these "custom" bridges:

[ ] Observe the "Link" Request: Look for the HTTP request that happens after the OAuth provider redirects back but before you are logged in. This is often a POST to a /register or /sync endpoint.

[ ] Inspect Hidden Defaults: Check if parameters like password, secret, or otp are being passed with values like null, none, default, or not-provided.

[ ] Test the "Traditional" Login: If you sign up with OAuth, always go back to the standard login form and try to log in with your email and the "placeholder" values found in step 1.

[ ] Analyze Custom Fields: Developers often add custom logic to standard OAuth flows to make them fit their legacy database. These "hand-off" points are the most likely place for bugs.





When the /v3/login endpoint receives your u parameter, the backend code (likely PHP or Go) checks it against a Whitelist.

If it's on the whitelist: (e.g., manager.infomaniak.com), it proceeds.

If it's NOT on the whitelist: (e.g., evil-attacker.com), the server will usually do one of two things:

Drop it: It ignores your input and resets u to the default dashboard.

Error: It stops the request entirely.

A. Parameter Pollution
Try sending the parameter twice. Sometimes the server validates the first one but the JWT-builder uses the second one.

GET /v3/login?u=https://manager.infomaniak.com&u=https://evil.com


Testing Checklist (Takeaways)
This is "Difficulty: High" because it requires finding abandoned assets. When hunting:

[ ] Audit Pre-Authorized Apps: Look for apps that don't ask for permission. Check their developer settings or associated domains.

[ ] Domain/Subdomain Takeover: If you see a redirect_uri pointing to a subdomain (e.g., dev-callback.example.com), check if that subdomain is pointing to a "404" on a service like AWS, GitHub Pages, or Heroku. If you can claim it, you can steal the tokens.

[ ] Check Legacy/Marketing Sites: Big companies often create small "campaign" sites for a product launch (e.g., win-a-car-contest.com). Once the contest ends, they forget to remove it from their OAuth whitelist.
